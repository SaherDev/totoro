# Implementation Plan: Gateway Per-User Rate Limiting

**Branch**: `016-gateway-rate-limit` | **Date**: 2026-04-19 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/016-gateway-rate-limit/spec.md`

## Summary

Add per-user rate limiting to the NestJS gateway via a new `RateLimitModule` (in-memory `Map<userId, UserRateLimitState>`), a `RateLimitGuard` applied to `POST /api/v1/chat`, and post-response counter updates in `ChatService` using two new fields FastAPI adds to every `ChatResponseDto` (`session_started: boolean`, `tool_calls_used: number`). The synchronous JSON response mode is retained — no SSE streaming. Also: rename `ClerkUser` → `AuthUser` (move to `libs/shared`, add `plan?: PlanTier`), extend the Clerk webhook to handle `session.ended` for turns reset, and add `rate_limits` config block to `app.yaml`.

## Key Design Notes

### Guard execution order on POST /api/v1/chat

```
ClerkMiddleware (global) → AiEnabledGuard (@RequiresAi) → RateLimitGuard → ChatController → ChatService
```

`RateLimitGuard` runs after `AiEnabledGuard`. Disabled users (ai_enabled: false) never consume quota.

### Counter update sequence in ChatService

```
Guard passes (limits not exceeded)
  → chatService.chat() called
  → rateLimitService.incrementTurns(userId)        // count this turn immediately
  → aiClient.chat(payload)                         // forward to FastAPI (sync JSON)
  → response = { ..., session_started?, tool_calls_used }
  → if (response.session_started) → rateLimitService.onSessionStarted(userId)
  → rateLimitService.addToolCalls(userId, response.tool_calls_used)
  → return response to frontend (unchanged)
```

`session_started` and `tool_calls_used` are consumed by NestJS for counter tracking. They are present in the returned `ChatResponseDto` but the frontend ignores them (forward-compatible per ADR-019).

Turns are incremented even if the FastAPI call fails (the turn was consumed). This is intentional.

### Day-reset logic (central method in RateLimitService)

The reset boundary is **UTC midnight** — always. The user's local timezone and the server's system timezone are both irrelevant. `new Date().toISOString()` always returns UTC regardless of where the Node.js process is running (`TZ` env var, Railway region, etc.). This is the only safe cross-timezone approach.

```typescript
private getTodayUtc(): string {
  // ISO string is always UTC: "2026-04-19T14:23:00.000Z" → "2026-04-19"
  // Safe regardless of server TZ env var or user locale.
  return new Date().toISOString().slice(0, 10);
}

private resetIfNewDay(counter: { count: number; date: string }): void {
  const today = this.getTodayUtc();
  if (counter.date !== today) {
    counter.count = 0;
    counter.date = today;
  }
}
```

**Never use**: `new Date().toLocaleDateString()`, `Date.prototype.getDate()`, `getMonth()`, `getFullYear()` — these use the process's local timezone and will produce the wrong date when the server runs in a non-UTC region.

Called at the start of every read or write on a daily counter (`sessions`, `toolCalls`).

### Lazy Map initialization

```typescript
private getOrCreate(userId: string): UserRateLimitState {
  if (!this.state.has(userId)) {
    const today = this.getTodayUtc();
    this.state.set(userId, {
      sessions:  { count: 0, date: today },
      toolCalls: { count: 0, date: today },
      turns: 0,
    });
  }
  return this.state.get(userId)!;
}
```
